# DataFlow v0.2 — Major Improvement Phase Specification

> **Created:** 6 April 2026
> **Baseline:** v0.1.0 — 770 tests, 45 test files, 17 DB tables, ~7200 LOC
> **Target:** ~2,000 tests, 100 improvements across 10 themed batches
> **Approach:** Layered Foundation — security-first, bottom-up dependency ordering

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Batch ordering | Security-first, layered | Fix foundations before building features on them |
| Client-side encryption | Full WebCrypto migration | True zero-knowledge: server never sees plaintext |
| External APIs | Allow HIBP (k-anonymity) | Privacy-safe breach checking, optional with fallback |
| Test target | ~2,000 tests | Near-triple coverage including edge cases |

## Batch 3: Client-Side Encryption

**Theme:** Migrate to true zero-knowledge architecture — server never sees plaintext.
**New tests:** ~130 | **Running total:** ~1,100

| # | Title | Description | Tests | Category |
|---|-------|-------------|-------|----------|
| 21 | WebCrypto key derivation | `public/js/crypto.js` — derive vault key from master password using PBKDF2 (WebCrypto) with 600,000 iterations + SHA-256. Output 256-bit AES key. Salt stored server-side per user. | 15 | crypto |
| 22 | Client-side AES-256-GCM | Encrypt/decrypt functions using `crypto.subtle.encrypt/decrypt` with AES-256-GCM. Random 12-byte IV per operation. Return `{ciphertext, iv, tag}` matching server format. | 15 | crypto |
| 23 | Key wrapping in browser | Wrap vault key with a key-encryption-key (derived from master password with different salt). Store wrapped key on server. Unwrap on unlock. | 12 | crypto |
| 24 | Encrypt fields before send | Item create/edit: encrypt `title`, `notes`, and all field values in browser before `POST /api/items`. Server stores ciphertext directly without re-encryption. | 15 | crypto |
| 25 | Decrypt fields after receive | `GET /api/items/:id` returns ciphertext. Browser decrypts using vault key held in memory. Vault list shows decrypted titles. | 12 | crypto |
| 26 | Server encryption removal | Remove server-side encrypt/decrypt from `item.service.js`. Server validates ciphertext format (base64, IV length, tag length) but never decrypts. `session-vault.js` no longer holds raw vault keys. | 10 | crypto |
| 27 | Data migration pathway | One-time migration endpoint `POST /api/data/migrate-encryption`: server decrypts all items with old key, returns plaintext batch to browser, browser re-encrypts with client key, PUTs back. Atomic per-item migration with progress tracking. | 15 | crypto |
| 28 | Client-side attachment encryption | Encrypt files in browser using `ReadableStream` + WebCrypto before upload. Decrypt after download. Update `public/js/crypto.js` with streaming encrypt/decrypt. | 12 | crypto |
| 29 | Sharing with client encryption | Implement X25519 key agreement: each user has a key pair. Sharing re-encrypts item with shared secret derived from sharer's private key + recipient's public key. Store public keys in `users` table. | 15 | crypto |
| 30 | Key rotation | "Rotate vault key" action: generate new vault key, re-encrypt all items in browser, update wrapped key on server. Progress indicator for large vaults. | 12 | crypto |

**Acceptance criteria:**
- Server logs and database contain zero plaintext item data
- Vault unlock derives key in browser, never sends master password to server
- Existing items migrate seamlessly on first unlock after upgrade
- Sharing works across different vault keys via key exchange
- File attachments encrypted/decrypted in browser with streaming

**Critical risks:**
- Browser tab close during migration loses data → save migration progress to `localStorage`
- WebCrypto not available in older browsers → feature-detect, require modern browser
- Performance with large vaults → batch operations, Web Worker for crypto
